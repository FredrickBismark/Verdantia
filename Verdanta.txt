# VERDANTA — Implementation Framework
# Garden Management Application with LLM-Curated Plant Intelligence

## PROJECT IDENTITY

- **Name:** Verdanta
- **License:** MIT
- **Language:** Python 3.11+ (backend), TypeScript (frontend)
- **Target Platform:** Linux Mint on Minisforum UM760 (Ryzen 7 6800H, 16-32GB RAM), but should run on any OS
- **Deployment:** Local-first, single-user, no cloud dependency required
- **Repository Structure:** Monorepo

---

## CORE PHILOSOPHY

1. Everything runs locally by default. Cloud APIs (weather, LLM) are optional enhancers with offline fallbacks.
2. The LLM is not a chatbot bolted on — it is the intelligence layer that curates plant data, generates schedules, and proactively advises.
3. Data transparency: users can always see what sources informed a recommendation and what the LLM was given as context.
4. No paywall, no subscription, no telemetry, no cloud accounts required.

SERVICE LAYER — IMPLEMENTATION DETAILS
LLM Service (services/llm_service.py)
This is the most critical service. It abstracts multiple LLM providers behind a common interface.

Supported providers:

Ollama (local, free, default) — uses Ollama HTTP API at http://localhost:11434
Anthropic (Claude models) — uses anthropic Python SDK natively
OpenAI — uses openai Python SDK
Venice.ai — uses OpenAI-compatible API at https://api.venice.ai/api/v1
OpenRouter — uses OpenAI-compatible API at https://openrouter.ai/api/v1
Custom — any OpenAI-compatible endpoint with user-provided base URL

class LLMProvider(str, Enum):
    OLLAMA = "ollama"
    ANTHROPIC = "anthropic"
    OPENAI = "openai"
    VENICE = "venice"
    OPENROUTER = "openrouter"
    CUSTOM = "custom"

class LLMConfig(BaseModel):
    provider: LLMProvider
    model: str
    api_key: str | None = None
    base_url: str | None = None
    temperature: float = 0.3
    max_tokens: int = 4096

class LLMClient(ABC):
    @abstractmethod
    async def generate(self, prompt: str, system: str | None = None,
                       images: list[str] | None = None,
                       response_format: str | None = None) -> str: ...
    @abstractmethod
    async def stream(self, prompt: str, system: str | None = None) -> AsyncIterator[str]: ...

Implementation classes:

OllamaClient — POST to /api/generate (non-chat) and /api/chat (chat mode)
AnthropicClient — native anthropic.AsyncAnthropic SDK, handles vision via content blocks
OpenAICompatibleClient — works for OpenAI, Venice, OpenRouter, Custom via openai.AsyncOpenAI with custom base_url
Factory function: create_llm_client(config: LLMConfig) -> LLMClient

Task-specific model resolution:
The service checks AppSettings for llm_task_overrides. If a specific task (e.g., dossier_curation, daily_chat, photo_diagnosis, proactive_alerts) has an override, use that provider/model. Otherwise fall back to the default.

Plant Curation Service (services/plant_curation_service.py)
This is the intelligence engine for building plant dossiers.

Pipeline when a user adds a new plant:

Gather raw data:

User-provided import data (CSV, JSON, or manual form entry)
OpenFarm API scrape (if enabled in settings)
LLM web search for local/regional data (if enabled and LLM supports search)
LLM's own training knowledge as fallback
Store all raw sources as PlantDataSource records with source_type, source_name, source_url, and raw_data JSON.

LLM curation step:

Construct a detailed prompt containing ALL raw sources + the user's garden context (zone, location, soil type)
Ask the LLM to synthesize a structured dossier with these sections:
Overview (botanical info, growth habit, maturity timeline)
Planting Instructions (localized to zone, indoor start, direct sow, transplant timing)
Soil Optimization (localized, references user's actual soil test data if available)
Watering Guide (frequency, amount, method, seasonal changes)
Fertilization Schedule
Pruning & Training (if applicable)
Harvest Guide (signs of readiness, method, storage)
Pest Management (region-specific, organic + conventional)
Disease Management (region-specific, prevention + treatment)
Companion Planting
Local Performance Notes (zone-specific cultivar data)
Each section has a confidence level: high, medium, low
Contradictions between sources are explicitly flagged with the conflicting claims, which sources made them, and the LLM's assessment
Store the curated dossier by populating PlantSpecies structured fields + creating DossierSection records for each section.

Refresh capability: Users can re-run curation at any time (e.g., new season, new soil test data, want updated local info). The pipeline re-gathers, re-curates, and updates.

The LLM curation prompt must instruct the model to return structured JSON matching a defined schema so it can be programmatically parsed and stored.

OpenFarm Service (services/openfarm_service.py)
Queries the OpenFarm REST API at https://openfarm.cc/api/v1/crops.

Search: GET /crops?filter={query}
Detail: GET /crops/{slug}
Normalize the response into a standard dict format compatible with the curation pipeline.
Handle API being down gracefully (it's a community project, not always reliable).
Weather Service (services/weather_service.py)
Uses the Open-Meteo API (free, no API key required).

Endpoints used:

Current weather: https://api.open-meteo.com/v1/forecast with current_weather=true
Forecast (up to 16 days): same endpoint with daily and hourly parameters
Historical: https://archive-api.open-meteo.com/v1/archive with date ranges (data back to 1940)
Variables to request: temperature_2m, temperature_2m_max, temperature_2m_min, relative_humidity_2m, precipitation_sum, wind_speed_10m, soil_temperature_0cm, soil_moisture_0_to_1cm, uv_index, cloud_cover

Background sync: APScheduler job runs every 6 hours (configurable) to fetch and store forecast data. Historical data is fetched on-demand or when a new garden is created.

Data blending priority for display:

Local IoT sensors (most accurate for microclimate)
Manual user entries
Open-Meteo API data
Computed values:

Growing Degree Days (GDD): accumulated from daily min/max temps against a base temperature
Frost date estimation: based on historical data for the location
USDA hardiness zone: derived from coordinates using historical min temp data
Calendar Service (services/calendar_service.py)
Auto-schedule generation:
Given a Planting and its associated PlantSpecies + Garden:

Look up average last spring frost and first fall frost for the zone
Calculate indoor seed start date (if applicable): last_frost - (days specified in species data)
Calculate transplant or direct sow date
Generate recurring care events based on species care instructions
Estimate harvest window from days-to-maturity + planting date
Return list of proposed CalendarEvent objects for user approval
Weather-responsive rescheduling:
A scheduled task compares upcoming CalendarEvent records (where weather_dependent=True) against the current forecast:

Frost warning + tender transplant scheduled → flag with "critical" priority, suggest delay
Heavy rain + scheduled watering → suggest skip
Extended drought + moisture-loving crops → suggest extra watering
Extreme heat + planned outdoor work → suggest early morning
These generate advisory notifications, not automatic changes. The user decides.

Filter system:
The API supports filtering calendar events by:

plant_id / planting_id (single or multiple)
event_type (single or multiple)
date_range (start, end)
status (completed, pending, overdue)
priority
source (manual, auto, llm)
weather_dependent
Garden Advisor Service (integrated into services/llm_service.py)
The advisor builds rich context before every LLM interaction:

Context assembly for a garden-level query:

- Garden: name, location, zone, soil profile
- Current weather + 14-day forecast summary
- All active plantings with current status
- Recent sensor readings (last 24-48 hours)
- Upcoming calendar events (next 14 days)
- Recent harvest data
- Any active alerts or flagged issues

Context assembly for a plant-specific query:
All of the above, PLUS:

Full dossier for the specific plant
Photo history for the plant
All harvest logs for the plant
Season-over-season comparison if multi-year data exists
Relevant LLM interaction history (previous advice given about this plant)
Proactive alerts (background task):
Run daily via APScheduler:

Assemble full garden context
Send to LLM with prompt: "Review this garden's current state. Identify any concerns, time-sensitive actions, or recommendations. Return as a JSON array of alerts with priority, category, and actionable description."
Store alerts for display on the dashboard
Photo diagnosis:
Accept a photo (base64) + planting context. Send to a vision-capable LLM with the prompt to diagnose potential issues (pests, diseases, nutrient deficiencies). Requires a vision-capable model — the service checks if the current model supports vision and falls back or warns if not.

Sensor Service (services/sensor_service.py)
MQTT integration:

Connect to a local Mosquitto broker (configurable host/port)
Subscribe to a configurable topic pattern (default: verdanta/sensors/#)
Expected message format: JSON with sensor_id, sensor_type, value, unit, optional location
Parse and store as SensorReading records
Manual entry:
API endpoint accepts the same data structure for users without IoT hardware who want to manually log thermometer readings or other observations.

Sensor management:

IMPORTANT IMPLEMENTATION NOTES
Use async throughout the backend. SQLAlchemy async sessions with aiosqlite. All httpx calls use AsyncClient. FastAPI async route handlers.

Error handling: Every external API call (Open-Meteo, OpenFarm, LLM providers) must have proper try/except with timeouts. The app must remain functional even if all external services are down.

The frontend proxies API calls through Vite's dev proxy in development. In production, the FastAPI app serves the built frontend static files.

Photo storage is local filesystem, not database blobs. Store only the path in the database.

LLM curation can be slow (30-120 seconds depending on model). Implement it as a background task with a status polling endpoint, or use SSE to stream progress to the frontend.

All LLM interactions are logged to llm_interactions for transparency and debugging. The user can view what context was sent and provide feedback.

The database should use WAL mode for SQLite to allow concurrent reads during writes: PRAGMA journal_mode=WAL;

API key storage: For the MVP, store in SQLite app_settings table. Encrypt at rest using a local key derived from a machine-specific identifier. This is not bank-grade security but prevents casual exposure.

CORS: In development, allow http://localhost:5173. In production (when serving frontend from FastAPI), CORS is not needed.

No authentication system. This is a local, single-user application. No login, no user accounts, no sessions.
